01 / problem

The data exists — it's just trapped in prose.

Every renal-cancer patient generates clinic letters, an operative note, a pathology report, a discharge summary, and years of surveillance visits. Turning that into a research database is done by hand, one chart at a time.

Manual abstraction into the 303-column form is slow, and the source notes actively fight back. Optical character recognition mangles clinical terms — clear cell renal cell carcinoma arrives as “Wear cel rena cel cancinoma.” Encounters appear out of chronological order. Histology hides inside the operative note rather than as its own report.

And the shorthand is a minefield for naïve parsing: 6/12 means in six months, not the 6th of December; TCU x1/52 means review in one week. A date parser reads these as values and silently corrupts the dataset.

What the pipeline is being measured against

The same charts have already been abstracted by a human into a parallel workbook. The pipeline's output is scored field-by-field against that human reference — so the study answers a concrete question: can the model match, or exceed, a trained human abstractor, and where does it disagree? Results across 35 patients are reported in section 07.

02 / architecture

One pass through six stages.

The model is used for exactly two jobs — recognising sections and reading fields. Everything that decides where data belongs and whether it is correct is ordinary, inspectable code.

1

Segment

Language model

A single model call labels every encounter — first visit, operative, histopathology, discharge, follow-up — and tags each with its date. It reads through OCR noise and pulls embedded pathology out of the operative note as its own section.

emits → sections[] { type, date, text }
2

Route

Deterministic

Code — not the model — decides placement. Fixed encounters map to their folders; each follow-up visit is assigned to the target interval (6- and 12-months) it falls closest to. A target with no nearby visit is left empty rather than forced.

emits → { folder → text } + routing audit
3

Extract

Language model

Each section is read by a folder-scoped prompt that asks only for that folder's fields. Clinical rules are built in: pathology overrides imaging; “partial or radical” consent boilerplate never creates a radical-surgery record; shorthand is decoded, not mistaken for dates.

emits → coded field values + contentious-field notes
4

Merge → JSON

Deterministic

Field values from every folder combine into one record, carrying a _routing trail (which visit went where, and why) and a _notes list of every judgement call.

emits → one JSON record per patient
5

Write back

Deterministicreview

Values land in the patient's row of the Excel form; follow-up day-counts are computed into helper columns; contentious cells receive a comment so a clinician can adjudicate. The human-verified original is never overwritten.

emits → filled workbook with ⚠ comments + day-counts
6

Evaluate

Deterministic

The record is scored field-by-field against the human-abstracted workbook — agreement and per-field metrics, with tolerance for number rounding and visit-vs-lab date gaps.

emits → agreement report (model vs human)
03 / the routing rule

Which visit is the “6-month” visit?

Clinic scheduling slips, and a note's own label can't be trusted — a visit headed “6-month follow-up” was really on day 249. So placement is decided by arithmetic: the day-count from surgery, matched to the nearest target.

Worked example — patient 001

Three post-operative visits. Each follow-up slot keeps the single visit closest to its target; visits that aren't closest to any target are dropped.

The same rule protects against over-filling. Patient 010 had only two visits, at 43 and 249 days — both nearest to the 6-month target. The pipeline assigns the 249-day visit to fu6, drops the 43-day visit, and correctly leaves the 12-month slot empty rather than stuffing it with a one-month visit. The day-count is written beside each follow-up so you can verify the placement at a glance.

06 / quality control

Built to be checked, not trusted blindly.

The output is designed for a clinician to audit quickly — the contentious 5% is surfaced, not hidden in the agreeing 95%.

Contentious-cell comments

Judgement calls, source conflicts and deliberate blanks each get a ⚠ comment on the exact cell — e.g. a positive margin, or two haemostatic agents recorded in a one-value field.

Day-count beside each visit

A helper column records days from surgery to each follow-up, so a reviewer can confirm the right visit was routed to the 6- and 12-month slots.

The original is sacred

The model fills a copy of the form. The human-verified workbook — the study's ground truth — is never modified, keeping the comparison clean.

Strongest domain: surgical fields

Operative notes and discharge summaries are structured and templated — the model handles them most reliably. Perioperative fields (folder 7) reached 89% agreement and 63% recall; surgical pathology fields (folder 5) reached 76% agreement and 53% recall.

These are also the most methodologically clean numbers: essentially no flowsheet data exists in operative or discharge notes, so every miss here is a genuine model error — not a structural absence.

08 / methodology

What was assumed, and what was measured.

The comparison is between two different inputs, not two different abstractors. Transparency about this distinction is essential to interpreting the numbers honestly.

The input asymmetry

The clinical fellow performed a full Epic chart review — reading clinic letters, operative notes, discharge summaries, pathology reports, and flowsheet data (vitals, lab results, medication records). The model received only the combined clinical note text for each patient. This asymmetry is the study's central variable, not a confound to control away: the goal is to quantify precisely what a notes-only LLM can and cannot recover.

The evaluation framework

Each of the 303 schema fields was compared across both abstractors. Cells where both returned null were excluded as uninformative. Agreement used exact-match for coded and integer fields, ±2% relative tolerance for continuous values, and ±7 days for dates. The resulting outcome for each cell was classified into one of five categories.

Cell outcome taxonomy

Every informative cell falls into exactly one category. The provenance class determines whether a miss counts against the model or is excluded from the primary metric.

agree

Both filled; values match within tolerance. Counts toward agreement% and recovery.

disagree

Both filled; values conflict. Requires adjudication. Counts against agreement%.

model-only

Model filled; fellow left blank. Possible genuine find or hallucination — sampled for review.

miss (note)

Fellow filled from notes; model missed. A true model error. Counts against recovery_note.

out-of-input

Fellow filled from flowsheet; model had no access. Excluded from recovery_note. Not a model error.

Provenance assumptions — what was classified as note vs. flowsheet

Each field was pre-classified at schema design time based on clinical knowledge of Epic documentation patterns — not validated empirically per chart. This assumption drives what counts as a model error vs. a structural absence.

Assumed, not validated

Provenance labels are static per-field assumptions applied uniformly across all 35 patients. Clinicians variably paste lab values into notes via Epic SmartPhrases — the observed 11% flowsheet-in-note rate on lab fields confirms this assumption is imperfect. A future refinement would validate provenance empirically per chart before scoring.

Model

Claude Sonnet 4.6 throughout — the same model for every patient, every folder. No temperature parameter (deprecated in this model family); default sampling used. Model consistency is a study integrity requirement: mixing models across patients would confound the LLM-vs-human comparison.

Cohort

35 patients with renal cell carcinoma undergoing partial (nephron-sparing) nephrectomy at NUS, surgery dates July–December 2025. Matched against 35 charts independently abstracted by a clinical fellow performing full Epic review. 4 additional AI-extracted patients had no fellow counterpart and were excluded from comparison.

Scoring

Exact match for coded and integer fields. Continuous fields: ±2% relative tolerance (e.g. creatinine 45 vs 46 μmol/L passes). Dates: ±7 days. Both-null cells excluded as uninformative. Field-level and folder-level breakdowns computed across all 35 matched patients simultaneously.
